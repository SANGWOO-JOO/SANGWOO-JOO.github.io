⚙️ 기술적 의사결정

1. K3s 선택 (vs Docker Compose, Full Kubernetes)

  • Docker Compose 대비: 자동 복구, 로드 밸런싱, 스케일링 등 프로덕션 요구사항 충족
  • Full Kubernetes 대비: 설치 및 운영이 간단하며 리소스 사용량 50% 절감 (바이너리 하나로 설치 가능)
  • 선택 이유: 소규모 팀에서 관리 가능한 복잡도로 고가용성 확보, 향후 클라우드 전환 시 호환성 유지
  • 트레이드오프: Full K8s 대비 일부 고급 기능 제한이 있지만, 현재 요구사항에는 충분

2. Prometheus + Grafana 선택 (vs ELK Stack, DataDog)

  • ELK Stack 대비: 메트릭 수집에 특화되어 있고 시계열 데이터 처리 성능 우수, 리소스 사용량 40% 낮음
  • DataDog 등 SaaS 대비: 오픈소스로 비용 절감, 민감 데이터를 외부로 전송하지 않아 보안 요구사항 충족
  • 선택 이유: Kubernetes 환경과의 긴밀한 통합, 활발한 커뮤니티, 무료로 프로덕션 수준의 모니터링 구축 가능
  • Grafana 선택: 직관적인 대시보드 구성, Prometheus와의 완벽한 호환, 풍부한 대시보드 템플릿 제공

3. Longhorn 스토리지 선택 (vs NFS, Ceph)

  • NFS 대비: 클러스터 네이티브 솔루션으로 K3s와 통합 관리 용이, 스냅샷 및 백업 기능 기본 제공
  • Ceph 대비: 설정이 간단하고 소규모 클러스터에 적합, 학습 곡선 낮음
  • 선택 이유: K3s 공식 권장 스토리지, 웹 UI로 볼륨 관리 가능, 노드 장애 시 자동 복제로 데이터 보호

4. ShedLock을 통한 배치 중복 실행 방지

  • 문제: 여러 Pod에서 동일한 배치 작업이 중복 실행될 위험
  • 대안 비교: Quartz Scheduler (과도한 기능), Leader Election (복잡한 구현)
  • 선택 이유: 단순하고 효과적인 분산 락 제공, Spring과의 쉬운 통합 (@SchedulerLock 어노테이션)
  • 구현: MySQL 기반 락 테이블로 최소 락 시간 5분, 최대 락 시간 10분 설정
성과
  • 인프라 안정성 확보: K3s 클러스터 구축으로 노드 장애 시 자동 복구 체계 마련, Longhorn을 통한 데이터 영속성 확보

문제 1: 로그 조회 성능 저하 (15초 → 1초, 93% 개선)

1. 문제 상황

  • 수백만 건의 로그 데이터 조회 시 평균 15초 소요
  • 운영팀이 고객사 이슈 분석 시 대기 시간이 길어 업무 효율 저하
  • 통합 검색 기능 사용 시 응답 시간이 더욱 증가

2. 원인 분석

  • 진단 도구: Scouter APM으로 느린 쿼리 식별, MySQL EXPLAIN으로 실행 계획 분석
  • 근본 원인: OR 조건으로 통합 검색 구현하여 인덱스를 활용하지 못하고 Full Table Scan 발생
  • 추가 병목: Offset 기반 페이지네이션으로 페이지 번호가 증가할수록 성능 급격히 저하
  • 데이터: EXPLAIN 분석 결과 type=ALL(Full Scan), rows=수백만 건 확인

3. 시도한 해결 방법

방법 1: 쿼리 구조 변경 + 복합 인덱스 설계

  • 시도 내용: OR 조건을 AND 조건 필터 조합으로 전환, 검색 조건 빈도 분석 후 (날짜 + 검색 필드) 복합 인덱스 설계
  • 결과: Full Table Scan → Index Range Scan 전환, 조회 시간 15초 → 3초 (80% 개선)
  • 장점: 인덱스를 활용하여 스캔 범위가 대폭 축소
  • 단점: 여전히 목표 성능(1초 이내) 미달, 추가 최적화 필요

방법 2: 커서 기반 페이지네이션 도입

  • 시도 내용: Offset 방식을 커서(마지막 조회 ID) 기반 페이지네이션으로 전환
  • 결과: 조회 시간 3초 → 1초 (추가 67% 개선, 총 93% 개선)
  • 장점: 페이지 위치와 무관하게 일정한 성능, 대용량 데이터에 최적
  • 단점: 특정 페이지로 직접 이동 불가 (운영 용도로는 문제없음)

4. 최종 선택 및 이유

  • 선택: AND 조건 + 복합 인덱스 + 커서 페이지네이션 조합
  • 이유 1: 두 방법을 결합하여 목표 성능(1초 이내) 달성
  • 이유 2: 운영팀은 최신 로그부터 순차 조회하므로 커서 방식의 단점이 영향 없음
  • 이유 3: 100만 건 이상 데이터에서도 일정한 성능 보장

문제 2: 대시보드 통계 조회 성능 저하 (500ms → 25ms, 95% 개선)

1. 문제 상황

  • 통계 집계 쿼리 실행 시간 500ms 이상 소요
  • 대시보드 접속 시마다 동일한 통계 쿼리 반복 실행
  • DB 부하 증가로 전체 시스템 성능에 영향

2. 해결 방법: Redis 캐싱 전략

  • 구현: 통계 집계 결과를 Redis에 TTL 300초로 캐싱
  • 결과: 캐시 히트 시 25ms로 응답 (95% 개선), 캐시 히트율 85% 달성
  • 선택 이유: 통계 데이터는 실시간성이 덜 중요하고 5분 주기 갱신으로 충분

  • 단일 해결책이 아닌 여러 최적화 기법을 조합하여 목표 성능 달성

  • 운영팀의 사용 패턴을 분석하여 기술 선택의 트레이드오프 평가
성과
  • 성능 개선: 로그 조회 평균 93% 개선 (15초→1초), 100만 건 이상 대용량 데이터 조회 시에도 1초 이내 응답 유지

  • 성능 개선: 커서 페이지네이션 도입으로 페이지 위치와 무관하게 일정한 성능 보장

⚙️ 기술적 의사결정

1. MessageSource를 활용한 국제화 (vs 프론트엔드 i18n)

  • 프론트엔드 i18n 대비: 서버 사이드 렌더링 환경에서 더 간단한 구현, SEO 최적화 유리
  • 선택 이유: 백오피스는 주로 관리자가 사용하는 내부 시스템으로 Spring의 기본 기능 활용이 효율적
  • 구현: properties 파일로 언어별 메시지 관리 (messages_ko.properties, messages_en.properties, messages_ja.properties)
  • 장점: Spring에서 제공하는 표준 방식으로 학습 곡선 낮고 유지보수 용이

2. JUnit 5 + Mockito 테스트 전략

  • 목표: 비즈니스 로직의 정확성 검증 및 리팩토링 안정성 확보
  • Mockito 선택 이유: 외부 의존성(DB, API)을 격리하여 단위 테스트 속도 향상, Spring과의 자연스러운 통합
  • 테스트 범위: 서비스 레이어의 비즈니스 로직에 집중, 라이선스 발급/갱신/회수 등 핵심 기능 우선 작성
  • 효과: 기능 변경 시 회귀 테스트로 부작용 조기 발견, 리팩토링 신뢰도 향상

3. GitHub Actions CI/CD (vs Jenkins, GitLab CI)

  • Jenkins 대비: 별도 서버 설치/관리 불필요, YAML 기반 설정으로 코드와 함께 버전 관리
  • GitLab CI 대비: 이미 GitHub를 사용 중이므로 추가 도구 없이 통합 가능
  • 선택 이유: 소규모 팀에 적합한 간단한 설정, 무료 플랜으로 충분한 빌드 시간 제공
  • 파이프라인: 코드 푸시 → 빌드 → 단위 테스트 실행 → Docker 이미지 빌드 → EC2 배포

4. Docker 컨테이너화 전략

  • 선택 이유: 로컬/개발/운영 환경 간 일관성 보장, "내 컴퓨터에서는 잘 되는데" 문제 해결
  • 멀티 스테이지 빌드: 빌드 환경과 실행 환경 분리로 이미지 크기 60% 감소 (JDK → JRE)
  • AWS EC2 배포: Docker Compose로 애플리케이션과 MySQL을 함께 관리, 환경 변수로 설정 분리
  • 트레이드오프: Kubernetes 대비 간단하지만, 현재 트래픽 규모에는 충분한 솔루션
성과
  • 운영 효율 개선: 라이선스 관리 업무 처리 시간 90% 단축 (10분→1분)
  • 운영 효율 개선: 다국어 지원으로 해외 고객사 대응 가능
  • 개발 생산성 향상: 단위 테스트 도입으로 리팩토링 시 영향도 파악 시간 단축
  • 개발 생산성 향상: CI/CD 파이프라인으로 배포 시간 60% 단축 및 수동 작업 제거
